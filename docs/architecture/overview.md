# 架构总览

## 总体设计

桌宠命令行助手围绕稳定事件协议组织。具体工具的适配器负责把命令行工具事件转换成统一桌宠事件。本地桥接服务负责存储和广播事件。桌面端根据桥接状态渲染二维动态桌宠和交互面板。

```text
命令行工具
  -> 适配器
  -> 统一桌宠事件协议
  -> 本地桥接服务
  -> 桌面端
  -> 二维动态角色、面板、通知
```

## 核心组件

### 桌面端

职责：

- 渲染透明置顶桌宠窗口。
- 渲染二维动态模型、表情和动作。
- 展示气泡、轻提示和弹窗。
- 提供点击面板，展示任务详情和设置。
- 把用户操作发送回本地桥接服务。

第一版建议技术：

- `Electron`。
- TypeScript。
- 网页套接字客户端。
- 网页二维动态模型运行时。
- `pnpm workspace`。
- Tailwind CSS。

### 本地桥接服务

职责：

- 接收适配器事件。
- 规范化事件。
- 维护会话状态。
- 向桌面端广播更新。
- 接收桌面端用户操作。
- 在可能时把操作路由回来源适配器。
- 保存最近会话历史和设置。

第一版建议通信方式：

- 本地超文本接口用于接收适配器事件。
- 本地网页套接字用于向桌面端推送状态。

### 适配器

职责：

- 对接一个命令行工具或事件来源。
- 把原生事件转换成统一桌宠事件。
- 声明支持的用户动作。
- 在工具支持时把用户动作返回给工具。

第一版适配器：

- 代码代理工具适配器，基于工具钩子接入。

未来适配器：

- 通用命令包装器：`termpet run <command>`。
- 其他代码代理工具。
- 包管理器。
- 版本控制工具。
- 测试工具。
- 容器工具。

### 协议包

职责：

- 定义共享事件类型。
- 定义状态枚举。
- 定义动作结构。
- 定义会话结构。
- 定义协议版本。
- 提供校验工具。

## 事件流

### 工具状态事件

```text
工具钩子载荷
  -> 适配器脚本
  -> 提交到本地事件接口
  -> 桥接服务规范化并存储事件
  -> 广播给桌面端
  -> 桌面端更新桌宠状态
```

### 用户动作事件

```text
用户在桌宠中点击允许或拒绝
  -> 桌面端发送动作
  -> 桥接服务校验动作
  -> 桥接服务路由到适配器动作处理器
  -> 适配器在支持时回应命令行工具
  -> 桥接服务发出动作结果事件
```

## 会话选择规则

第一版默认规则：

- 主桌宠跟随最近活跃会话。
- 新事件到达时，对应会话成为活跃会话。
- `waiting_approval` 和 `error` 可以在提醒队列里覆盖普通状态。

多个会话接近同时更新时，提醒优先级建议：

```text
waiting_approval > error > working > thinking > success > listening > idle
```

产品第一版仍以最近活动为准，优先级主要用于提醒队列。

## 运行边界

桌面端不直接解析具体工具载荷。

适配器不直接控制二维动态模型运行时。

桥接服务负责会话状态和动作路由。

协议包负责共享结构和校验。

## 失败处理

- 如果桥接服务离线，适配器脚本应快速失败，不能阻塞命令行工具。
- 如果桌面端离线，桥接服务仍然保留有限数量的最近事件。
- 如果动作无法路由，桌面端应提示用户需要回到终端操作。
- 如果二维动态模型加载失败，桌面端应降级为简单静态状态视图。
