# 项目规则

## 文档语言规则

本项目所有说明文档统一使用中文。

适用范围：

- 根目录说明文档。
- `docs/` 下的产品需求文档。
- `docs/` 下的架构文档。
- 路线图、贡献说明、设计说明。
- 面向用户或开发者阅读的项目说明。

允许保留原文的内容：

- 项目名。
- 文件路径。
- 命令。
- 代码标识符。
- 协议字段。
- 第三方工具或框架名称。

不允许的内容：

- 大段英文说明。
- 用英文替代可以自然表达的中文解释。
- 中英文混排导致阅读重心偏向英文。

## 项目范围规则

第一版目标是实现代码代理工具的状态可视化桌宠。

第一版必须聚焦：

- 桌面桌宠窗口。
- 二维动态角色状态切换。
- 本地事件桥接服务。
- 代码代理工具事件适配。
- 点击面板。
- 多会话最新活动展示。
- 等待确认、失败、完成等关键状态打断提醒。

第一版暂缓：

- 自动识别所有终端输出。
- 多命令行工具适配。
- 插件市场。
- 大模型驱动角色灵魂。
- 远程会话同步。
- 聊天界面。

## 架构规则

桌面端不直接理解具体命令行工具。

正确边界：

```text
命令行工具
  -> 适配器
  -> 统一事件协议
  -> 本地桥接服务
  -> 桌面桌宠
```

任何新适配器都应该把工具事件转换为统一事件协议，而不是让桌面端为每个工具写特殊逻辑。

## 技术选型规则

第一版技术选型：

- 包管理器统一使用 `pnpm`。
- 工作区使用 `pnpm workspace`。
- 桌面端使用 `Electron`。
- 主要实现语言使用 TypeScript。
- 有界面样式时使用 Tailwind CSS。
- 不要在第一版同时维护多个桌面端技术方案。

## Git 提交规则

- 提交信息使用 Conventional Commits 格式，例如 `feat(scope): message`。
- 提交前必须通过类型检查。
- 推送前必须通过构建。
- 本仓库使用 `.githooks` 作为 Git hooks 目录，初始化后需要执行 `pnpm prepare:git`。
