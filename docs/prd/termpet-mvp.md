# 桌宠命令行助手第一版产品需求

## 摘要

桌宠命令行助手是一个桌面二维动态桌宠，用于表达命令行工具和代码代理的运行状态。第一版聚焦代码代理工具：接收生命周期事件，转换成统一状态协议，并驱动桌宠表达状态、主动提醒用户、展示任务详情。

## 目标

- 让用户不用盯着终端，也能快速感知代码代理当前状态。
- 使用二维动态角色的表情和动作表达任务状态。
- 当任务需要确认、失败或完成时主动提醒用户。
- 点击桌宠后展示当前任务、最近日志、快捷操作和设置入口。
- 支持多个会话，默认展示最近活跃会话。
- 为后续适配更多命令行工具预留插件架构。

## 目标用户

- 经常使用代码代理工具的开发者。
- 经常运行长时间命令的命令行重度用户。
- 喜欢桌宠形式，但希望工具具备实际效率价值的用户。

## 用户故事

1. 作为代码代理用户，我希望代理开始思考时桌宠切换表情，这样我知道任务已经开始处理。
2. 作为代码代理用户，我希望代理执行工具时桌宠展示工作动画，这样我知道它正在动手处理。
3. 作为代码代理用户，我希望需要权限确认时桌宠明显提醒我，这样我不会错过确认请求。
4. 作为代码代理用户，我希望能在桌宠上允许或拒绝请求，这样我不用频繁切回终端。
5. 作为代码代理用户，我希望任务失败时桌宠明确提示并展示最近日志，这样我能快速判断是否需要查看终端。
6. 作为多项目用户，我希望桌宠默认跟随最近活跃会话，同时还能在面板里切换其他会话。

## 功能需求

### 桌宠窗口

- 背景透明。
- 无边框。
- 置顶显示。
- 可拖拽。
- 重启后记住位置。
- 能根据状态切换二维动态模型的表情和动作。
- 至少支持以下状态：
  - `idle`：空闲。
  - `listening`：监听。
  - `thinking`：思考。
  - `working`：工作。
  - `waiting_approval`：等待确认。
  - `success`：成功。
  - `error`：失败。

### 状态打断

桌宠支持四类打断级别：

| 状态 | 默认打断级别 |
| --- | --- |
| `idle` | `silent` |
| `listening` | `bubble` |
| `thinking` | `bubble` |
| `working` | `bubble` |
| `success` | `toast` |
| `error` | `modal` |
| `waiting_approval` | `modal` |

第一版需要允许用户在设置中降低或关闭部分提醒。

### 点击面板

点击桌宠打开紧凑面板。

面板包含：

- 当前活跃会话。
- 当前状态。
- 任务标题和摘要。
- 最近日志。
- 可用快捷操作。
- 会话切换。
- 设置入口。

第一版不包含聊天输入框。

### 多会话行为

- 每个事件必须包含 `sessionId`。
- 主桌宠状态默认跟随最近活跃会话。
- 面板可以列出最近会话。
- 后续可以支持手动固定某个会话，第一版先以最近活动为准。

### 代码代理适配器

代码代理适配器负责消费工具钩子事件，并转换成桌宠事件。

重要生命周期：

- `SessionStart`
- `UserPromptSubmit`
- `PreToolUse`
- `PostToolUse`
- `PermissionRequest`
- `Stop`

映射关系：

| 工具事件 | 桌宠状态 |
| --- | --- |
| `SessionStart` | `idle` 或 `listening` |
| `UserPromptSubmit` | `thinking` |
| `PreToolUse` | `working` |
| `PermissionRequest` | `waiting_approval` |
| `PostToolUse` 成功 | `working` |
| `PostToolUse` 失败 | `error` |
| `Stop` 成功 | `success` |

### 确认流程

理想流程：

```text
代码代理发起权限请求
  -> 桌宠展示确认弹窗
  -> 用户点击允许或拒绝
  -> 适配器把决定返回给代码代理
```

如果第一版无法直接外部确认，则使用降级流程：

```text
代码代理发起权限请求
  -> 桌宠展示确认弹窗
  -> 明确提示用户回到终端确认
  -> 桌宠保持等待状态，直到代码代理继续
```

确认界面必须显示：

- 来源工具。
- 工作目录。
- 请求动作。
- 风险级别。
- 查看详情按钮。
- 在支持时显示允许和拒绝按钮。

## 非功能需求

- 钩子处理必须很快，不能阻塞代码代理执行。
- 本地桥接服务需要保留有限数量的最近事件。
- 敏感原始输出默认不展示。
- 即使桥接服务重启，桌面端也应能恢复连接。
- 桥接服务需要容忍重复事件。
- 事件协议需要版本号。
- 所有面向用户的确认动作都需要留下审计记录。

## 成功标准

第一版成功的标准：

- 运行代码代理时，桌宠能正确切换状态。
- 至少一个二维动态模型能表达所有核心状态。
- 等待确认、失败、完成等状态能被用户明显感知。
- 点击桌宠能看到当前会话和最近日志。
- 多会话能被跟踪，主桌宠跟随最近活跃会话。
- 代码边界清晰：协议、桥接、桌面端、适配器各自独立。
