## Context

当前桥接服务已经提供 `/state`、`/sessions`、`/events` 和 WebSocket `snapshot` / `event` 消息，桌面端也已经接入了部分真实数据流。但 `PRD-02` 还要求完整的主会话切换、桥接断线降级、恢复重连和去除演示态依赖，这些内容横跨桥接客户端、渲染状态和桌宠展示组件，适合先固定设计再继续实现。

## Goals / Non-Goals

**Goals:**

- 让桌面端以单一状态源消费桥接服务，而不是混合真实状态与演示态。
- 明确“初始化拉取、实时订阅、断线重连、轻量降级”的状态流。
- 让后续 `PRD-03` 和 `PRD-04` 可以直接建立在稳定的实时状态基线上。

**Non-Goals:**

- 不在本次 change 中实现点击面板完整交互。
- 不在本次 change 中实现等待确认弹窗和关键状态打断提醒。
- 不扩展到除代码代理之外的其他命令行工具。

## Decisions

### 1. 保持桥接服务为唯一事实来源

桌面端初始化时先请求桥接快照，再通过 WebSocket 接收 `snapshot` / `event` 增量消息。这样可以保证应用重启和桥接重启后都能恢复一致状态，比继续保留本地 `demoState` 更稳定。

备选方案：

- 保留演示态作为兜底。未采用，因为这会让真实状态和演示状态并存，难以判断问题来源，也不符合 `PRD-02` 验收标准。

### 2. 在渲染层集中管理连接状态和桌宠展示状态

桥接连接状态、当前活跃会话状态和最近事件应在渲染层状态容器中集中维护，再由 `pet-shell` 等组件消费。这样可以避免组件直接处理网络事件，降低后续接入提醒面板和多会话视图的复杂度。

备选方案：

- 由组件各自订阅桥接事件。未采用，因为会造成状态分散，不利于后续扩展。

### 3. 断线时降级显示，不销毁窗口

桥接断开时，桌宠应进入轻量降级状态，保留窗口与基本交互，并持续尝试重连。这样符合 `PRD-02` 的“断线不影响窗口继续存在”要求。

备选方案：

- 断线后直接显示空白或要求用户重启。未采用，因为用户体验差，也不利于长时间运行。

## Risks / Trade-offs

- [桥接状态和渲染状态映射不一致] -> 通过单一状态容器和固定映射函数减少分叉。
- [断线重连频率过高造成噪声] -> 使用轻量重连策略和明确的连接状态标记，避免频繁抖动。
- [后续多会话面板接入时重复造状态结构] -> 当前就把活跃会话和会话列表的边界拆清楚，为 `PRD-04` 复用。

## Migration Plan

- 本次变更属于本地开发态增强，不涉及线上数据迁移。
- 实施顺序为先调整桌面端状态容器，再补连接恢复和渲染降级，最后通过桥接模拟验证。
- 如果实现中发现当前桥接消息结构不足，再回到 PRD 和 OpenSpec 补充说明后继续改代码。
