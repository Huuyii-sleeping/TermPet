## Context

第一版功能链路已经基本完成，但应用仍缺少三类关键收口：桌面端设置持久化、桥接端历史与审计持久化、以及异常降级时的明确反馈。`PRD-06` 的目标不是扩功能，而是把现有 MVP 补到“可以持续使用”。

## Goals / Non-Goals

**Goals:**

- 应用重启后恢复窗口位置和提醒偏好。
- 本地保存有限的最近事件、最近会话和动作审计。
- 让用户能够在桌宠面板中查看基础设置和最近审计摘要。
- 在桥接离线和角色仍处于占位模式时给出清晰降级说明。

**Non-Goals:**

- 不实现云同步。
- 不实现完整插件安装管理界面。
- 不实现大模型驱动角色灵魂。

## Decisions

### 1. 桥接数据使用本地 JSON 文件持久化

桥接服务运行在本地 Node 环境中，当前最小可用方案是把最近事件、会话和审计记录以 JSON 形式落到本地目录，并设置上限。这样实现简单、可调试，也符合“原始数据只本地保存”的约束。

### 2. 桌面端设置与主窗口位置分别持久化

窗口位置由主进程持久化到 `userData` 目录，因为它属于 Electron 窗口层状态；提醒偏好由渲染层本地存储持久化，因为它只影响前端提醒展示逻辑。

### 3. 第一版审计展示保持轻量

桥接端提供审计查询接口，桌面端在面板设置区展示最近几条审计摘要即可，不扩展到完整日志检索界面。

## Risks / Trade-offs

- [本地 JSON 文件损坏导致状态恢复失败] -> 读取时做容错回退，失败则回到空状态。
- [桥接频繁写盘影响性能] -> 只在事件和动作变更时写入，且严格控制保留上限。
- [设置项过多使面板过重] -> 第一版只保留提醒强度与成功提醒开关两项。

## Migration Plan

- 先补协议中的审计记录类型和桥接持久化结构。
- 再补主进程窗口位置恢复和渲染层设置 hook。
- 最后接入面板设置区与审计摘要展示，并通过本地模拟验证。
