## Context

目前 Codex 适配器已经能把基础状态映射成标准事件，桌面端也已经具备关键提醒 UI，但动作按钮仍然只是展示，没有真正的桥接动作接口。`PRD-05` 需要在不引入完整插件运行时和真实远程回传的前提下，先把第一版最小动作链路接上。

## Goals / Non-Goals

**Goals:**

- 细化 Codex hook 事件的状态映射和摘要信息。
- 为权限确认等关键状态生成统一动作列表。
- 增加桥接服务的动作请求入口和结果返回模型。
- 让桌面端能发起动作请求并获得明确的下一步提示。

**Non-Goals:**

- 不实现其他命令行工具适配器。
- 不实现真实 approve / deny 回传到 Codex 进程。
- 不实现完整插件市场和插件运行沙箱。

## Decisions

### 1. 动作请求先走桥接服务统一入口

桌面端不直接理解每种工具动作的回传细节，而是将动作请求发给桥接服务，由桥接服务做最小校验和路由。这样符合当前架构边界，也为后续插件化执行留接口。

### 2. 当前版本优先返回“可执行”或“终端回退”结果

由于第一版还没有真实回传能力，桥接服务对 `approve`、`deny`、`rerun`、`stop` 等动作统一返回带终端回退提示的结果；`dismiss` 和 `open_detail` 可以在本地直接处理。

### 3. Codex 映射使用保守启发式

对 `Stop`、`PostToolUse`、错误和中断类事件，适配器基于 hook 名称、成功标记、退出码和停止原因做保守判断。目标是让第一版“常见生命周期事件稳定落标准状态”，而不是追求对所有未来 hook 变体的完全覆盖。

## Risks / Trade-offs

- [启发式映射覆盖不全] -> 优先覆盖常见成功、失败、人工中断三类路径，保留默认兜底。
- [动作接口还不能真实执行] -> 返回明确的终端回退提示，避免用户误以为桌宠内已执行。
- [动作按钮引入更多状态分支] -> 将动作结果收束到统一结果结构，桌面端只做提示显示。

## Migration Plan

- 先补协议和插件 SDK 的动作类型来源。
- 再增强 Codex 事件映射。
- 然后增加桥接 `/actions` 路由与桌面端调用。
- 最后通过本地模拟验证动作请求和结果提示。
