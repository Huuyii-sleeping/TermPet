## Context

`PRD-02` 已经完成桌面端真实状态同步，桌宠可以被桥接服务实时驱动。`PRD-03` 要在这条状态流之上增加主动提醒能力，覆盖等待确认、失败和完成三类关键事件，同时保持第一版范围，不实现真正的动作回传和完整设置系统。

## Goals / Non-Goals

**Goals:**

- 让桌宠能根据事件的 `interruptLevel` 渲染最小提醒层级。
- 让 `waiting_approval` 和 `error` 事件以强提醒方式显示，并且保留必要信息与详情入口。
- 让 `success` 事件以轻提醒方式短暂展示，不长期阻塞桌面。
- 在无法直接执行确认动作时，明确提示用户返回终端处理。

**Non-Goals:**

- 不实现真实的 approve / deny / rerun / stop 动作回传。
- 不实现完整设置面板和提醒强度配置。
- 不在本次变更中补完整的多会话切换界面。

## Decisions

### 1. 提醒层直接消费活跃会话的最近关键事件

当前桌宠主视图已经由 `useBridgeState` 驱动，本次提醒层继续建立在同一状态源之上，只关注活跃会话最近事件。这样能减少新的状态分叉，也更符合第一版“多会话默认以最近活跃会话驱动主桌宠状态”的原则。

备选方案：

- 为所有会话维护并发提醒队列。未采用，因为这会提前引入复杂的多会话提醒仲裁，超出 `PRD-03` 范围。

### 2. 使用前端本地提醒状态管理 `toast` 的生命周期

`modal` 和 `bubble` 可以直接绑定当前关键事件状态，但 `success` 的 `toast` 需要短时显示后自动消退，因此在前端本地维护一个轻量提醒状态更合适。

备选方案：

- 把提醒生命周期放回桥接服务。未采用，因为第一版还不需要跨端共享提醒展示状态。

### 3. 强提醒信息优先展示事实，不隐藏风险信息

`waiting_approval` 与 `error` 的提醒卡片必须直接展示来源工具、工作目录、动作摘要和详情入口；如果当前事件无法直接执行动作，就明确给出终端回退提示。这比仅展示拟人化文案更符合产品安全边界。

备选方案：

- 仅用桌宠气泡文案表达风险。未采用，因为容易让用户错过关键信息。

## Risks / Trade-offs

- [成功提示过于频繁造成打扰] -> 让 `toast` 自动消退，并限制为轻量样式。
- [强提醒遮挡桌宠主视图] -> 强提醒只在关键状态触发，内容控制在必要信息范围内。
- [Codex 事件细节不足] -> 在适配器侧补最小元数据和终端回退动作描述，为后续真实动作链路预留接口。

## Migration Plan

- 先补 Codex 适配器的关键提醒字段。
- 再在桌面端加入提醒状态与最小 UI。
- 最后通过模拟关键状态事件验证 `waiting_approval`、`error`、`success` 的表现。

## Open Questions

- 当前版本的“详情入口”先使用卡片内展开摘要，还是额外弹出独立面板。当前倾向先用卡片内展开，避免提前引入完整面板系统。
