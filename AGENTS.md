# 项目规则

这份文件是本项目给代码代理和自动化协作者的长期规则。进入本仓库后，应优先阅读并遵守本文件。

## 文档语言

本项目的所有说明文档必须使用中文撰写。

要求：

- 新增或修改 `README.md`、`docs/` 下文档、产品需求、架构说明、路线图、贡献说明时，正文必须使用中文。
- 不要整段使用英文说明。
- 技术名词、目录名、包名、命令、代码标识符、文件路径、协议字段可以保留原文。
- 如果需要解释英文技术名词，应优先给出中文说明。

## 命名约定

- 用户可见的中文名称暂定为“桌宠命令行助手”。
- 代码、包名、协议命名可以使用 `termpet`、`TermPet`、`termpet`。
- 文档中可以保留产品代号，但解释性正文优先使用中文名称。

## 产品方向

本项目第一阶段聚焦代码代理工具的桌宠状态可视化。

长期方向是做成可插拔的命令行工具状态桌宠：

- 先支持代码代理工具。
- 后续支持更多命令行工具适配器。
- 桌宠通过统一事件协议表达不同工具的状态。
- 角色系统包含模型资源和角色灵魂配置。

## 第一版范围

第一版必须聚焦：

- 桌面桌宠窗口。
- 二维动态角色状态切换。
- 本地事件桥接服务。
- 代码代理工具事件适配。
- 点击面板。
- 多会话最新活动展示。
- 等待确认、失败、完成等关键状态打断提醒。

第一版暂缓：

- 自动识别所有终端输出。
- 多命令行工具适配。
- 插件市场。
- 大模型驱动角色灵魂。
- 远程会话同步。
- 聊天界面。

实现时可以为这些后续能力预留接口，但不要把它们做成第一版交付范围。

## 技术选型

- 包管理器统一使用 `pnpm`。
- 第一版桌面端统一使用 `Electron`。
- 工作区优先使用 `pnpm workspace` 组织。
- 桌面端、桥接服务、协议包、插件开发接口优先使用 TypeScript。
- 有界面样式时统一使用 Tailwind CSS。
- 不要在第一版引入其他桌面壳方案作为并行实现。
- 后续如果要评估更轻量桌面方案，必须先形成文档决策，再改实现。

## 架构边界

必须保持以下边界：

```text
命令行工具
  -> 适配器
  -> 统一事件协议
  -> 本地桥接服务
  -> 桌面桌宠
```

要求：

- 桌面端不直接解析具体命令行工具的原始载荷。
- 适配器负责把工具事件转换成统一事件协议。
- 本地桥接服务负责会话状态、事件广播、动作路由和最近历史。
- 协议包负责共享类型、版本和校验。
- 二维动态模型运行时只消费标准状态，不直接理解工具事件。

## 插件和适配器规则

- 第一版可以内置代码代理适配器，但代码结构要能迁移到插件。
- 新命令行工具接入时，优先使用钩子型适配器。
- 没有钩子能力时，优先考虑 `termpet run <command>` 包装器。
- 自动解析终端输出属于后续能力，不作为第一版默认方案。
- 用户自己开发的命令行工具如果暴露钩子，应允许通过适配器接入。

## 事件协议规则

- 所有状态事件必须包含协议版本。
- 所有状态事件必须包含 `sessionId`。
- 事件摘要默认短小，适合气泡或通知展示。
- 原始载荷、大段日志、代码片段、密钥和文件内容不要直接放进普通事件摘要。
- 多会话默认以最近活跃会话驱动主桌宠状态。

## 安全与隐私规则

- 敏感数据默认隐藏。
- 原始日志和原始载荷只能本地保存，并设置保留上限。
- 等待确认、拒绝、允许、停止、重跑等动作需要可审计。
- 中高风险动作必须显示来源工具、工作目录、动作摘要、风险级别和详情入口。
- 未经用户明确开启，不要把原始日志、命令输出、源码或权限请求详情发送给大模型。
- 本地桥接服务默认只绑定 `127.0.0.1`。

## 角色灵魂规则

- 第一版角色灵魂先使用规则配置，不接入大模型生成行为。
- 角色灵魂可以影响文案、动作、表情、打断频率和表达强度。
- 角色灵魂不能改变事件事实、严重级别或安全提示。
- 文案应同时兼顾主要信息和简短拟人化表达。

## 前端体验规则

- 桌宠是工具界面，不要做成营销落地页。
- 第一屏应直接是可用桌宠或桌宠控制界面。
- 点击桌宠展示任务详情、最近日志、快捷操作和设置，不提供聊天输入框。
- 重要事件打断要清晰，但需要可配置，避免长期使用时过度打扰。
- 权限确认界面必须明确、克制、可读，不要把风险信息藏在装饰性视觉里。

## 协作要求

- 文档修改要保持产品需求、架构、协议、路线图之间的一致性。
- 新增功能前优先检查 `docs/` 下已有设计。
- 不要把第一版范围扩大到所有命令行工具；第一版先实现代码代理可视化。
- 修改协议、插件接口、安全规则时，同步更新相关文档。
- 如果实现发现现有文档不可行，应先更新文档里的决策和限制，再继续改代码。
- 每次完成实现后，需要向用户补充一段中文说明，解释本次是如何实现的，至少说明修改了哪些模块、关键数据流或状态流如何变化、主要函数或组件分别承担什么职责。

## Git 提交规则

- 提交信息必须使用 Conventional Commits。
- 示例：`feat(workspace): 初始化项目骨架`。
- 提交前必须通过 `pnpm typecheck`。
- 推送前必须通过 `pnpm build`。
- 仓库初始化后执行 `pnpm prepare:git`，启用 `.githooks`。
